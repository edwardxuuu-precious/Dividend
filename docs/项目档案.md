## 1. 项目目标

构建一个本地运行的 A 股分红率（股息率）监控仪表盘：

1. 第一阶段：盯紧贵州茅台 `600519`，盘中实时看到股息率随价格变化。
2. 第二阶段：扩展到几十只长期分红股票，做横向对比与估值判断。
3. 长期：成为个人定投/价投的辅助决策工具——不止"看数据"，要能"指出贵贱"。

设计原则：

- **配置即扩展**：加股票只改 `config/watchlist.yaml`，业务代码不动。
- **数据源可替换**：抽 `PriceSource` / `DividendSource` / `HistoricalPriceSource` 三个接口，akshare 只是其中一个实现。
- **静态/动态分离**：历史数据缓存重用，实时数据每次重算；前后端表里如一。
- **不引入构建工具/框架**：FastAPI + 单文件 HTML/JS，跑得起来就够用。

### 3.5 历史分位阈值（估值锚点）
- **目的**：自动判断当前股息率"在历史上属于贵还是便宜"。高股息率 = 股价相对便宜。
- **实现**：
  - 后端：[app/services/history_service.py](../app/services/history_service.py) `compute_percentiles()`、`percentile_rank()`、`valuation_label()`
  - 算法：仅取 `source == window` 的真实数据样本（剔除 carry 兜底估计、lapsed/pre_first 的 0 值）→ 线性插值得到 P10/25/50/75/90 → 当前值用二分查找定位排名
  - 前端：图表上叠加 P25/50/75/90 横虚线（颜色区分），右侧标注阈值
- **验收**：
  - 茅台当前 3.54% 落在 P91，标签"历史性低估"
  - P90 阈值 3.49%（即历史上 90% 时间股息率低于 3.49%）
  - 5 个分类标签（深度低估/偏低估/中性/偏高估/深度高估）阈值清晰、无歧义
  - 方案 D 上线后分位数与之前完全一致（茅台 P25/P50/P75/P90 = 0.88/1.28/2.31/3.49 不变），证明 carry 段未污染样本
- **状态**：✅ 已交付

### 3.6 明年分红预估（三档）
- **目的**：为长期持有者估算下年度分红现金流，结合现价反推预期股息率。
- **实现**：
  - 后端：[app/services/history_service.py](../app/services/history_service.py) `compute_annual_history()` + `forecast_next_year()`
  - 三档：
    - **保守** = 与去年持平
    - **中位** = 去年 × (1 + 近 3 年平均 YoY) ← 推荐档
    - **乐观** = 去年 × (1 + 近 3 年最大 YoY)
  - 置信度：连续派现年数 ≥5 高 / ≥3 中 / 否则低
- **验收**：
  - 茅台 2026 预估：保守 51.63 / 中位 54.94 / 乐观 62.80（分别对应按现价 3.54% / 3.77% / 4.31%）
  - 中位档蓝边高亮 + "推荐"小徽章
  - 单测 `tests/test_percentile_and_forecast.py` 8 个用例
- **状态**：✅ 已交付

### 3.7 盘中实时同步（关键功能）
- **目的**：避免"首页表格在跳、详情面板冻结 6h"的体验割裂；盘中点开详情看到的就是此刻数字。
- **设计**：
  - HistoryService 拆为静态/动态：
    - **静态**（series / percentiles / annual / forecast / EOD）→ 6h 文件缓存
    - **动态**（current）→ 永远基于最新 quote 实时计算，**不缓存**
  - 新增轻量端点 `GET /api/yields/{symbol}/current`，仅返回 ~10 个字段，几百字节
  - 前端打开详情面板后，每 `refresh_seconds` 秒（默认 10s）poll 一次该端点
- **实现**：
  - 后端：[app/services/history_service.py](../app/services/history_service.py) 拆 `_get_static()` + `_compute_live_current()`
  - 接口：[app/main.py](../app/main.py) `GET /api/yields/{symbol}/current`
  - 前端：[app/web/static/app.js](../app/web/static/app.js) `pollLiveCurrent()` + `applyLiveCurrent()` + chart 的 `updateLive()` handle
  - 共享 PriceService 实例：watcher 与 history_service 共用同一个 5s 内存缓存，避免重复请求新浪
- **图表标记**：
  - **EOD dot**（蓝色小点）= 上一个交易日收盘对应 TTM
  - **Live dot**（橙色大点 + 脉冲动画）= 此刻实时股息率，与 EOD 之间用虚线连接表示"当日变动"
- **验收**：
  - 盘中：首页"现价 / 股息率"与详情面板顶部大字、徽章 P 分位、live dot 位置完全同步
  - 闭市：live dot 与 EOD dot 重叠（live source 自动退化为 EOD）
  - 测试覆盖：[tests/test_live_current.py](../tests/test_live_current.py) 4 个用例
    - 实时价驱动 yield 重算
    - 报价缺失退化到 EOD
    - 不同价格 → 不同 yield + percentile_rank
- **状态**：✅ 已交付（2026-04-26）

### 3.12 完整年度判定（dividend_year 修复）
- **目的**：解决 §3.11 暴露的横向对比失真——已派 2026 中期分红的股票（长电/格力）在主表显示的是不完整的当年金额（长电 0.21 / 格力 1.00），yield 严重偏低。
- **实现**：
  - 新增共享 helper [app/services/dividend_service.py](../app/services/dividend_service.py) `annual_payment_groups()` + `drop_incomplete_latest_year()`
  - **启发式**：最新年的派息次数 < 前 3 年中**任一年**的最大派息次数 → 视为进行中、剔除。用 max（而非众数）能捕捉模式变化（如长电 2025 起从 1 次/年改为 2 次/年）。
  - 历史不足 3 年时不做判断（避免误伤新股）
  - 同时用于 [app/services/history_service.py](../app/services/history_service.py) `compute_annual_history()`，让年度表 + forecast 都用完整年
- **验收**：
  - 长电：修前 year=2026 div=0.21 yield=0.78%；修后 year=2025 div=0.943 yield=3.52%（P50 中性）
  - 格力：修前 year=2026 div=1.00 yield=2.69%；修后 year=2025 div=3.00 yield=8.08%（P95 历史性低估）
  - 招行：保持 year=2026 div=1.013 yield=2.57%（**正确**：招行每年只 1 次年度分红，2026 年付的就是 FY2025 的完整年度）
  - 茅台：保持 year=2025 div=51.63 yield=3.54%（2025 年内 2 次都已实现）
  - 单测 [tests/test_dividend_service.py](../tests/test_dividend_service.py) 加 3 个 case：incomplete latest 跳过 / complete latest 保留 / 历史不足不判断
  - 单测 [tests/test_percentile_and_forecast.py](../tests/test_percentile_and_forecast.py) 加 1 个 case 覆盖 compute_annual_history
- **状态**：✅ 已交付（2026-04-26）

### 3.15 日 K 增量缓存
- **目的**：日 K 24h TTL 到期后原本会全量重抓（茅台 5907 bar / 7 只 ≈ 7-10s）；现在只补差量（每天 0-3 个新交易日，毫秒级）。

### 3.16 TTM 假 0 谷修复（方案 D）+ 停止分红警报
- **目的**：消除 TTM 走势图上周期性出现的"假 0"谷——A 股大多数公司一年只分 1 次红，且每年除权日漂移（同一公司今年 5 月、明年 7 月很常见）；只要相邻除权日间隔 > 365 天，严格滚动窗口在中间空窗期就归零，曲线被拉到 0% 谷底，视觉上误导。同时捕捉真正的"公司停止分红"信号（如格力 2017-2019 那次"董明珠不分红"事件）。
- **三档判定（方案 D）**：
  - **window** —— 365 天滑动窗口里有真实分红，TTM = 窗口内累计（与原算法一致）
  - **carry** —— 窗口为 0 但距上次除权 ≤ `carry_stale_days`（默认 540 天 ≈ 18 个月），TTM = 上一笔除权所在自然年度累计（兜底估计，避免假 0）
  - **lapsed** —— 距上次除权 > 540 天，视为公司已停止分红，TTM = 0（触发警报）
  - **pre_first** —— 公司首次除权之前，TTM = 0
- **实现**：
  - 后端：[app/services/history_service.py](../app/services/history_service.py)
    - `compute_ttm_series(bars, events, *, carry_stale_days)` 三档逻辑；series 元素由 4 列扩为 5 列 `[date, close, ttm, yield_pct, source]`
    - `summarize_lapsed(series, events)` 输出 `{currently_lapsed, days_since_last_ex, last_ex_date, historical_lapsed_count, stale_threshold_days, segments[]}`，写入 history payload；`segments` 每段含 `start_date / end_date / days / prev_ex_date / resumed_ex_date / ongoing`，供前端徽章可点开查看明细
    - `_meaningful_yields()` 仅取 `source == window` 的天，确保 carry/lapsed 不污染分位样本
    - 缓存名 v2 → v3（payload 加 `lapsed_summary` 字段，旧缓存自然失效）；2026-04-28 再升 v8 → v9：`lapsed_summary.segments` 新增
  - 配置：[app/config.py](../app/config.py) `AppConfig.carry_stale_days: int = 540`；[config/watchlist.yaml](../config/watchlist.yaml) 顶层可覆盖；[app/main.py](../app/main.py) 透传给 `HistoryService`
  - 前端：[app/web/static/app.js](../app/web/static/app.js)
    - `buildChart()` 把 series 按 source 拆段：window 段实线、carry 段虚线（端点共享衔接）、lapsed/pre_first 段不画线（断开）
    - tooltip 在非 window 点显示 source 标签（"兜底估计" / "已停止分红" / "公司未分红期"）
    - 详情顶部 `.lapsed-badge` 徽章：`currently_lapsed=true` → 红底白字 "⚠ 已停止分红 (距上次除权 X 天，约 Y 个月)"；`historical_lapsed_count>0` → 橙色 "历史曾断流 X 次"
    - 徽章为 `<button>`，点击展开 `.lapsed-detail-panel` 明细面板（`renderLapsedDetailPanel()`）：列出每段 `第 N 段 / 起止日期 / 跨度天数 / 触发：上一次 ex_date 后超过 540 天 / 恢复：下一次 ex_date 或"⚠ 至今未恢复"`；ongoing 段红色高亮；面板顶部说明判定规则，底部注明"与折线图断点一一对应"作为数据 backup

  - 样式：[app/web/static/app.css](../app/web/static/app.css) 新增 `.chart-line-carry`（虚线，opacity 0.55）、`.lapsed-current`、`.lapsed-history`；2026-04-28 再补 `.lapsed-clickable`/`.lapsed-caret`/`.lapsed-detail-panel`/`.lapsed-detail-row`（含 `.is-ongoing` 红边）等明细面板样式

## 4. 数据流与缓存策略

| 数据 | 缓存类型 | TTL | 何时穿透 | 实现 |
|---|---|---|---|---|
| 当前价 | 内存 (单实例) | 5s | 首次或过期 | `app/services/price_service.py` |
| 历史分红事件 | 文件 `data/dividend_cache.json` | 24h | 首次或过期 | `app/services/dividend_service.py` |
| 历史日 K | 文件 `data/daily_bars_cache.json` | 24h | 首次或过期 | `app/services/history_service.py` |
| 历史静态部分 (series + 分位 + 年度 + 预估 + EOD + lapsed_summary) | 文件 `data/history_static_v3_cache.json` | 6h | 首次或过期 | `app/services/history_service.py` |
| **实时 current** | **不缓存** | — | 每次请求都算 | `app/services/history_service.py:_compute_live_current` |
| 失败兜底 | 文件 `get_stale()` | 永久 | 数据源异常时 | `app/cache.py:FileCache.get_stale` |

### 端点行为

```
GET /api/yields                  → watchlist 全部行情（5s 行情缓存 + 24h 分红缓存）
GET /api/yields/{s}/history      → 完整 payload（静态 6h 缓存 + 动态实时 current）
GET /api/yields/{s}/current      → 仅动态部分（实时算）
GET /api/health                  → 存活检查
```

### 前端轮询节奏

```
启动 → 立即 GET /api/yields
      ↓ 每 refresh_seconds 秒（默认 10s）
      GET /api/yields  …

点击行展开 → GET /api/yields/{s}/history（一次，含初始 current + 静态全部）
            启动 setInterval 每 refresh_seconds 秒：
              GET /api/yields/{s}/current  →  applyLiveCurrent()
点击行收起 → AbortController 取消 + clearInterval
```

## 5. 数据源说明

### 行情（实时）

**新浪 `https://hq.sinajs.cn/list=...`**（直连，不走 akshare 包装）

- 选它的原因：
  - 东方财富 `82.push2.eastmoney.com` 在某些 Windows 客户端下会强制 TLS 重协商后断流（schannel 兼容问题），不可靠
  - akshare `stock_zh_a_spot()` 分页 60+ 次拉全市场，慢且浪费
  - 新浪批量接口：一次请求拿整个 watchlist，毫秒级
- 编码：`sh600519` / `sz000858` / `bj430047`
- 解析：`var hq_str_sh600519="名称,开,昨收,现价,..."`，现价取 fields[3]

### 历史分红事件

**akshare `stock_history_dividend_detail(symbol, indicator="分红")`**（东财）

- 列：`公告日期 | 送股 | 转增 | 派息(每10股) | 进度 | 除权除息日 | 股权登记日 | 红股上市日`
- 关键转换：`派息 / 10` 得每股；`进度` 为"实施"才算（"预案"跳过）
- `除权除息日` 类型是 `datetime.date` 不是 `datetime.datetime`（曾经踩过坑，见早期任务记录）

### 历史日 K

**akshare `stock_zh_a_daily(symbol, adjust="")`**（新浪）

- 不复权 — 股息率必须用真实价格
- 选它而非 `stock_zh_a_hist`：东财日 K 端点同样有 `RemoteDisconnected` 问题
- 列：`date / open / high / low / close / volume / amount / outstanding_share / turnover`

### 系统代理处理

A 股数据源都在国内，不应走 Clash/V2ray 这类代理：

```python
# app/main.py 启动时清除：
import os
for v in ("HTTP_PROXY", "HTTPS_PROXY", "http_proxy", "https_proxy", "ALL_PROXY", "all_proxy"):
    os.environ.pop(v, None)

# Windows 系统代理（注册表 IE 设置）也要 patch：
import urllib.request
urllib.request.getproxies = lambda: {}
```

不这么做会触发 SSL `DECRYPTION_FAILED_OR_BAD_RECORD_MAC`。
